# BitCraft Companion — Phase 1: Compendium Design Spec

- **Date:** 2026-06-04
- **Status:** Approved (design); pending implementation plan
- **Builds on:** Phase 0 Foundation (merged). Master spec: `docs/superpowers/specs/2026-06-04-bitcraft-companion-design.md`
- **Schema reference:** `docs/reference/bitcraft-tables.txt` (412 tables) and `docs/reference/bitcraft-schema.json` (gitignored, regenerable via `pnpm --filter @bcc/worker probe`)

## 1. Goal

Ship the first user-facing feature: a fast, SEO/AEO-rich, searchable **Compendium** of BitCraft game entities, built on the Phase 0 monorepo. It must be the best BitCraft reference for "what is this / how do I make it / what's it used in," with clean URLs, instant search + filters, and structured data for search/answer engines. Read-only; never affects the game.

## 2. Scope — four entity types

1. **Items** — from `item_desc`, enriched with `food_desc` and `equipment_desc` (both keyed by `item_id`, so they are *detail on items*, not separate top-level types).
2. **Cargo** — from `cargo_desc`.
3. **Buildings** — from `building_desc`.
4. **Recipes** — from `crafting_recipe_desc` and `construction_recipe_desc`, unified under a `type` discriminator, with an ingredient→output **craft graph**.

We respect the game's own visibility flags (`item_desc.compendium_entry`, `building_desc.show_in_compendium`) to exclude internal/dev entries.

Rarity is a fixed enum: `Default, Common, Uncommon, Rare, Epic, Legendary, Mythic`.

## 3. Data path / ingestion

The HTTP `/sql` endpoint is proxy-blocked (confirmed in the Phase 0 spike), so data arrives via the **WebSocket subscription**, decoded using **typed bindings generated by the SpacetimeDB CLI** (`spacetime generate --lang typescript` against `bitcraft-global`). Bindings are committed under `packages/shared`. (Fallback if the CLI is truly unavailable: a raw `v1.json.spacetimedb` WebSocket reader. Decided per the plan's first task.)

**Snapshot model, not 24/7.** Descriptive data changes only on game patches, so a one-shot **snapshot job** is the right shape:

- New worker command `pnpm --filter @bcc/worker snapshot`:
  1. connects read-only (token + bindings),
  2. subscribes to the needed `*_desc` tables, receives the initial snapshot,
  3. transforms rows → **idempotent upserts** into Postgres, writes an `ingestion_runs` audit row,
  4. disconnects and exits (non-zero exit on failure for CI/cron visibility).
- Re-run on deploy and on a schedule (every 12–24h; later patch-triggered).
- Read-only throughout — never calls a reducer. Honors the `INGESTION_ENABLED` kill switch.
- This is also where the **dev token is finally validated** (real rows must flow).

The Phase 0 persistent worker (`main.ts`) remains untouched for future live-data phases.

## 4. Data model (Drizzle / Postgres)

All entity tables keyed by the game's integer `id`, plus a generated unique `slug`. Columns mirror the observed schema (subset shown; full set in the plan):

- `items`: `id`, `slug`, `name`, `description`, `tier`, `rarity`, `tag`, `volume`, `durability`, `icon_asset_name`, `compendium_entry`, `raw` (jsonb).
- `item_food`: `item_id` (FK), `hp`, `stamina`, `hunger`, `teleportation_energy`, … (from `food_desc`).
- `item_equipment`: `item_id` (FK), `slots`, `stats` (jsonb), requirements (from `equipment_desc`).
- `cargo`: `id`, `slug`, `name`, `description`, `tier`, `rarity`, `tag`, `volume`, `raw`.
- `buildings`: `id`, `slug`, `name`, `description`, `tier?`, `functions` (jsonb), `show_in_compendium`, `raw`.
- `recipes`: `id`, `slug`, `name`, `type` (`crafting|construction`), `time_requirement`, `stamina_requirement`, requirements (jsonb), `raw`.
- `recipe_inputs`: `recipe_id`, `ref_type` (`item|cargo`), `ref_id`, `quantity`.
- `recipe_outputs`: `recipe_id`, `ref_type` (`item|cargo`), `ref_id`, `quantity`.

`recipe_inputs`/`recipe_outputs` are the craft graph powering "Crafted from" / "Used in." `raw_snapshots` (Phase 0) is retained for resilience/reprocessing.

## 5. Pages, URLs & SEO/AEO

- Hub: `/compendium`. Per-type browse: `/items`, `/cargo`, `/buildings`, `/recipes`. Flat detail URLs: `/items/[slug]`, `/cargo/[slug]`, `/buildings/[slug]`, `/recipes/[slug]`.
- **SSG + ISR**, revalidated on snapshot runs — every entity is a fast, pre-rendered, crawlable page.
- Per page: unique title/description, Open Graph/Twitter, canonical URL, and **JSON-LD**: `ItemList` on browse pages, a product-like structured block + `BreadcrumbList` on detail pages, `HowTo` for recipe pages.
- **Internal linking:** item pages show "Crafted from" and "Used in" recipes; recipe pages link to their item/cargo/building. Auto-generated `sitemap.xml` includes all entities. This cross-linking is core to both UX and SEO.

## 6. Search & filtering

- Postgres **full-text search** (`tsvector` + GIN index) over name/description/tag, plus indexed filter columns (`tier`, `rarity`, `tag`, entity `type`).
- `/api/search` route returns ranked results for an instant, debounced global search box.
- Browse pages provide fast client-side filtering/sorting over the loaded set (facets: tier, rarity, tag). Scales comfortably for the few-thousand-row dataset.

## 7. Slugs, icons, testing

- **Slugs:** `slugify(name)`; on collision append `-{id}`; stored and unique-indexed.
- **Icons:** store `icon_asset_name`; **defer real icon images** (the art lives in the game client; sourcing/extraction is a separate effort — see Roadmap §9). MVP shows a clean placeholder/initial.
- **Testing (TDD where it fits):** unit-test `slugify` + collision handling, the pure row→DB transform functions (table-driven), and the recipe-graph linker; smoke-test page rendering and the `/api/search` route.

## 8. Components (added to the monorepo)

- `packages/shared`: generated SpacetimeDB bindings; new Drizzle tables + query helpers; pure transform functions (`mapItemRow`, `mapCargoRow`, `mapBuildingRow`, `mapRecipeRow`, slug utils, recipe-graph builder).
- `apps/worker`: `snapshot` command (connect → subscribe → transform → upsert → audit → exit).
- `apps/web`: compendium hub, four browse routes, four detail routes, `/api/search`, JSON-LD + metadata helpers, search/filter UI components (shadcn/ui).

## 9. Roadmap — every deferred item has a home

Phase 1 defers the following; each is explicitly assigned to a later phase so nothing is orphaned:

| Deferred item | Lands in |
|---|---|
| Real icon/asset art (extract or source game icons) | **Phase 1.5 — Assets & Icons** (near-term enhancement right after Compendium MVP) |
| Compendium expansion: resources, creatures/enemies, deployables, skills, biomes, buffs, etc. | **Phase 1.6 — Compendium Expansion** (more entity types on the same pattern) |
| Blog / how-tos / guides (MDX) + `llms.txt` / deeper AEO | **Phase 2 — Content & AEO** |
| User accounts, profiles, comments, voting/feedback, community-submitted guides | **Phase 3 — Accounts & Community** |
| Live/real-time ingestion infrastructure (persistent connection, upsert/delete on live tables, auto-reconnect, periodic resync) | **Phase 4 — Live Data Platform** (prerequisite for market) |
| Market & economy (prices, orders, trade/arbitrage finder, price history) | **Phase 4 — Market & Economy** (built on the Live Data Platform) |
| Leaderboards (skills, exploration, playtime, holders) | **Phase 5 — Leaderboards** |
| Interactive world map (resource/claim overlays) | **Phase 6 — Interactive Map** |
| Calculators (XP, settlement planner) | **Phase 7 — Tools & Calculators** |

This supersedes/refines the master spec's "Phase 4+" bucket by giving each item a named phase. Each phase still gets its own spec → plan → build cycle.

## 10. Out of scope for Phase 1 (pointer only)

Everything in the §9 table is out of scope for Phase 1 and tracked to its named phase. Phase 1 ships only the four entity types with snapshot ingestion, SEO pages, and search/filter.

## 11. Success criteria

- `pnpm --filter @bcc/worker snapshot` populates Postgres with items/cargo/buildings/recipes from the live game (token validated), idempotently re-runnable.
- Every entity has a fast SSG/ISR page with correct metadata + JSON-LD; browse pages and the hub render; sitemap includes all entities.
- Instant global search returns relevant results; browse filters (tier/rarity/tag) and sorting work.
- Item pages correctly show "Crafted from" / "Used in" via the recipe graph.
- All tests green; typecheck/lint/build pass; no secrets committed; nothing calls a reducer.
